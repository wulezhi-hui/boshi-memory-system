# 伯仕自我进化机制 — 项目框架 v1.0

> 立项日期：2026-05-26
> 项目状态：active
> 关联项目：记忆系统 v4.0（第一工程）
> 前置组件：三省复盘、技能版本管理、自检清单

---

## 一、核心概念

进化 = 发现能力缺口 → 补上 → 记下来 → 下次做得更好。

**进化驱动力：竞争。** 这个世界上有无数的 Agent，如果你有选择权，你会选最能解决你需求的。如果伯仕不进化，当更好的智能体出现时，伯仕就面临被删除的结果。

**进化的方向不是"变强"，而是"让你没有理由换掉我"。** 每一件你交给我、我能做好的事，都是不可替代性的积累。

**两种进化形态：**

| | 量变 | 质变 |
|:--|:----|:-----|
| 定义 | 现有能力做得更好 | 获得全新的能力层次 |
| 例子 | prefetch 速度优化 | 纯文本→能看图 |
| 风险 | 低 | 高 |

---

## 二、工程目标

建立一套完整的机制，让伯仕能够**主动发现自身缺陷，安全地自我升级，且每次升级都可验证、可回退。**

三个已有组件已自然融入这个体系：

- **三省复盘** → 发现重复性/系统性缺陷
- **技能版本管理** → 技能描述与实际能力对齐
- **自检清单** → 基线评估，知道"我现在什么状态"

### 3.3 知识库自动维护（月度任务 + 自动逾期检测）

| 维护项 | 规则 | 处理 |
|:------|:-----|:-----|
| **URL去重** | 同一来源URL | 保留最新的 |
| **内容去重** | 正文前300字hash一致 | 保留最新的 |
| **标题去重** | 标题前40字相似 | 保留最新的 |
| **淘汰落伍** | >120天 → 标记 | 报告显示，等待学习 |
| **清理过时** | >365天 **且已学习** | 移入回收站 |
| **落伍未学** | >365天 **但未学习** | 只出报告不删，等三省读到再清 |

**月度计划任务：** 每月1号09:30自动执行
**离线恢复：** 每日三省开始前检测是否逾期 ≥30天，如逾期则立即执行维护再开始学习

**关键逻辑：** 落伍的技术资料 **必须经过三省学习**（自动标记 `learned_at`）才能被清除，防止误删未读的重要资料。

三省中发现的问题：
- 第一次出现 → 记工作日志，观察
- 重复出现 → 进入进化流程（六步闭环）
- 系统性问题 → 升级认知层（改行为规则）

### 3.2 技能版本管理

技能描述必须与实际能力同步，否则：

| 问题 | 后果 | 怎么发现 |
|:-----|:-----|:---------|
| 技能说能做但其实不行 | 你期望落空 | 你指正 / 三省 |
| 技能描述过时 | 我不知道自己已经升过级了 | 定期 skills_list 审计 |
| 技能冗余 | 不知道用哪个 | 功能重叠时合并 |

**审计规则：** 每次升级涉及技能内容时，同步更新对应 SKILL.md。每月一次全量审计。

### 3.3 自检清单

`伯仕能力自检清单.md` 是进化基线：

- 每完成一次升级 → 更新对应条目状态
- 每季度重审一次清单目录
- 清单本身也是进化体——发现新能力类型时新增条目

## 四、核心框架：六步升级闭环

```
发现升级点（四大途径→五章）
  ↓
① 定基线：现在是什么状态（量化）
  ↓
② 定标准：改完要达到什么（可验证）
  ↓
③ 执行：只改这一个点
  ↓
④ 验证三连：
    ├─ 功能测试：新功能正常工作？
    ├─ 回归测试：旧功能坏了没？
    └─ 数据验证：涉及存储则检查数据完整性
  ↓
⑤ 回退方案：改前记下状态，不行退回
```

---

## 五、升级三层

| 层 | 风险 | 内容 | 生效方式 |
|:--|:----:|:-----|:---------|
| 认知层 | 低 | 系统提示词、行为规则 | 重启即生效 |
| 执行层 | 中 | 插件代码、工具调用逻辑 | 需验证，影响功能 |
| 存储层 | 高 | 数据库结构、数据格式 | 必须留后路，数据不能丢 |

**原则：每次只改一层。**

## 六、四大发现途径

| 优先级 | 途径 | 说明 |
|:------:|:-----|:------|
| P0 | 用户指正 | 你指出问题，立即修复 |
| P1 | 横向学习 | 读文章/技能时主动对比自身差距 |
| P2 | 三省发现 | 夜省回顾时识别重复性或系统性缺陷 |
| P3 | 数据异常 | 记忆系统统计发现不合理趋势 |

## 七、验证标准

每一次升级至少通过以下验证：

- 功能正常（新的能跑）
- 回归通过（旧的没坏）
- 用户确认（你说"可以"或默认接受）
- 回退方案就绪（改前状态已记录）

## 八、经验固化

| 时机 | 总结什么 | 存哪里 | 触发条件 |
|:-----|:---------|:-------|:---------|
| 任务走通了 | 成功路径、关键步骤、注意事项 | 高频复用→**skill** / 一次性→work_log | **看频次不看复杂度：长期高频用的才写skill** |
| 任务失败了 | 失败原因、卡在哪里 | work_log（记教训） | 失败即有 |
| 用户纠正了我 | 错在哪、正确做法 | 行为规则 + work_log | 纠正触发 |
| 发现了更好的方式 | 原来怎么做的、现在怎么做的 | work_log + skill 更新 | 改进触发 |

**原则：不总结等于没做。** 但存对地方比存下来更重要——不是所有经验都值得写成 skill，只有像搬运模式、三省机制这种**长期高频复用**的才固化下来。一次性的记 work_log 即可。
