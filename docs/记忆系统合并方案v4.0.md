# 记忆系统合并方案 v4.0 — 统一存储+热度调度

> 制定日期：2026-05-26
> 当前版本：v3.2（多存储隔离）
> 目标版本：v4.0（统一 Chroma + 热度调度）

---

## 一、架构变更总览

### 改前（v3.2 — 三套物理存储）
```
hot.json（热区话题） ← 独立文件
ChromaDB（温区事实/日志） ← 独立
cold.json（冷区归档） ← 独立
state.db（全量原始对话） ← 不动
```

### 改后（v4.0 — 统一 Chroma + 热度标签）
```
ChromaDB（全部记忆） ← 统一存储
  ├── tier:hot    热度≥10，自动加载
  ├── tier:warm   热度1~10，boshi_search命中
  ├── tier:cold   热度<1，boshi_excavate挖掘
  ├── type:conversation  对话摘要
  ├── type:raw_conversation  完整原始对话
  ├── type:work_log  工作日志
  └── type:project_log  项目日志
state.db（Hermes原始对话） ← 兜底，不动
```

## 二、存储统一

### 2.1 写入策略

每轮对话后同步写 Chroma：

```python
def sync_turn(user_content, assistant_content):
    # 1. 写入完整原始对话（新增）
    add_memory(
        content=f"用户: {user_content}\n伯仕: {assistant_content}",
        metadata={"type": "raw_conversation", "tier": "warm"}
    )
    
    # 2. 写入提炼摘要（已有）
    add_memory(
        content=f"用户: {user_content[:200]} | 伯仕: {assistant_content[:200]}",
        metadata={"type": "conversation_digest", "tier": "warm"}
    )
    
    # 3. 热度管理
    检测已有话题
        匹配 → heat += 10, mention_count += 1, last_mentioned = now()
        不匹配 → 新建话题, heat = 20
```

### 2.2 迁移计划：从 hot.json / cold.json 合并到 Chroma

| 数据源 | 条目数 | 目标类型 | 迁移方式 |
|:-------|:------:|:---------|:---------|
| hot.json | ~21条热话题 | tier:hot | 提取话题摘要+heat值，写入 Chroma |
| Chroma已有 | ~892条 | 保留原有 tier | 已有数据只需加 heat 字段 |
| cold.json | 少量归档 | tier:cold | 直接写入 |

步骤：
1. 读取 hot.json → 每条话题转为 Chroma 条目，带 heat + tier:hot
2. 读取 cold.json → 写入 Chroma，标记 tier:cold
3. 确认 Chroma 包含全部数据后 → 标记 hot.json / cold.json 为只读
4. 观察一周无异常 → 删除 hot.json / cold.json

### 2.3 热度字段规范

每条 Chroma 条目 metadata 中新增以下字段：

| 字段 | 类型 | 说明 |
|:-----|:----|:------|
| `heat` | float | 热度值 0.0~100.0 |
| `tier` | string | `hot` / `warm` / `cold` |
| `mention_count` | int | 被提及次数 |
| `last_mentioned` | timestamp | 最近一次提及时间 |
| `type` | string | 条目类型 |

## 三、热度调度机制

### 3.1 热度更新规则

```
被提及 → heat += 10, 上限 100
每天衰减 00:00 → heat *= 0.9
heat ≥ 10 → tier = 'hot'
1 ≤ heat < 10 → tier = 'warm'
heat < 1 → tier = 'cold'
```

### 3.2 prefetch 读取规则

```
新对话开始 → prefetch()
    ↓
查 Chroma:
  WHERE tier = 'hot'
  ORDER BY heat DESC, last_mentioned DESC
  LIMIT 20
    ↓
注入前 10 条到上下文
剩下 10 条做备选

boshi_search 搜索时:
  先查 tier:hot → 再查 tier:warm → 最后 tier:cold
  结果按 heat DESC 排序
```

### 3.3 冷区回收

```
每天 02:00 cron:
  查 Chroma WHERE heat < 1 AND last_mentioned < 7天前
  → 标记 tier = 'cold'
  → 压缩 content 长度到 200 字以内
```

## 四、实施步骤

### Phase 1：改写入（改 sync_turn）
- 添加完整原始对话写入（type: raw_conversation）
- 添加热度更新逻辑
- 保持现有摘要写入不变

### Phase 2：改读取（改 prefetch + boshi_search）
- prefetch 改为查 Chroma tier:hot 排序返回
- boshi_search 结果按 heat 降序排列
- 移除 hot.json 读取代码

### Phase 3：迁移历史数据（一次性）
- hot.json → Chroma（带 heat 值）
- cold.json → Chroma
- 验证数据完整性

### Phase 4：收尾
- hot.json 标记只读
- cold.json 标记只读
- 观察一周后删除

## 五、风险与回退

| 风险 | 应对 |
|:-----|:------|
| Chroma 查询变慢 | 保持 LIMIT 控制，不超过 30 条 |
| 热度衰减过头 | heat 下限设为 0.1，不死透 |
| 迁移数据丢失 | hot.json 改为只读不删除，保留 7 天 |
| 查询结果太多 | prefetch 只取 top 10 进上下文 |
