## 二、记忆层内部架构

### 2.1 写入链路

```
所有通道对话
    ↓
sync_turn() ← 每轮对话后自动触发
    ├→ store_conversation() → tiered_memory (热区/温区)
    ├→ push_live_turn() → 跨通道实时同步
    ├→ extract_facts() → Chroma (关键事实提取)
    ├→ 强制写入对话摘要 → Chroma (conversation_turn类型)
    └→ _auto_worklog() → Chroma (自动检测代码变更)
```

特点：
- 写入是异步的（daemon 线程），不阻塞对话
- 写入覆盖所有通道：CLI / 微信 / 工作台 / WebUI
- 自动工作日志检测 patch/write_file/skill_manage 操作

### 2.2 读取链路

```
新会话启动
    ↓
initialize() → queue_prefetch() → 后台预取
    ├→ 热区话题 (tiered_memory get_context_injection)
    ├→ 跨通道最近2天对话 (state.db 查询)
    ├→ 项目上下文 (检测关键词→加载 project_log)
    ├→ 语义关联搜索 (Chroma query)
    └→ 冷区兜底 (温区<3条时自动 excavate)

手动搜索
    ├→ boshi_search(query) → Chroma 语义搜索 (支持 where 过滤)
    ├→ boshi_excavate(query) → 冷区深度挖掘
    ├→ session_search(query) → state.db 全文检索
    └→ _direct_query_recent_sessions() → 同步 state.db 查询
```

### 2.3 存储层

| 存储 | 内容 | 大小 | 访问方式 |
|:----|:-----|:----:|:--------|
| ChromaDB | 工作日志、项目日志、对话摘要、联想知识、关键事实 | ~892条 | 语义搜索 (boshi_search) |
| tiered_memory | 热区话题、温区记忆、冷区归档 | 分3层(热/温/冷) | 自动加载 + 手动搜索 |
| state.db (全量) | 所有通道原始对话 — 第4层 | 全量 | SQL查询 (session_search) |
| Obsidian知识库 | 技术文章、UE5笔记 | 手动管理 | 文件系统 (ls) |

## 四、行为规则（系统提示词内）

每次对话前自动注入到上下文，共5条：

① 用户问事务性问题 → 先查证再回答
② 用户是问句（非指令）→ 只回答，不动手
③ 用户说'开始/做吧/执行' → 才动手
④ 提到项目/文章/记忆系统 → 先查 project_log / work_log
⑤ 没有叫你执行前 → ❌ 不修改代码/文件，只回答

## 五、技术支持

### 5.1 搜索过滤 (where 参数)

`boshi_search(query, where={"type": "project_log"})` 支持 metadata 过滤：
- 单条件：`{"type": "project_registry"}`
- 多条件：`{"$and": [{"type": "project_log"}, {"project": "项目名"}]}`

### 5.2 自动工作日志检测

检测范围：
- patch 操作 → 记录修改文件名 + diff 变更行
- write_file 操作 → 记录新建文件名 + 字节数
- skill_manage 操作 → 记录技能操作名

去重机制：
- 同文件 60 秒内不重复写
- 搜索已有条目比对新内容

### 5.3 跨通道记忆接续

- 新会话自动加载最近 2 天所有通道的对话摘要
- 实时写入（sync_turn），不需要手动保存
- boshi_search 跨通道语义搜索
