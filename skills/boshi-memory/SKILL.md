# 伯仕记忆系统 (BoShi Memory)

记忆模块路径：`~/.hermes/memory/boshi_memory.py`
存储位置：`~/.hermes/memory/`（随 Hermes 配置迁移）

工作台路径：`~/.hermes/workstation/伯仕工作台.pyw`
工作台地址：`http://localhost:7681`

## 每次对话的标准流程

收到用户消息后：

### Step 0: 理解需求，直接做
**最重要的一条规则：不要问乐之技术方案选择。**
- 他说"把工作台搞好" → 直接干
- 他说"怎么回事" → 先查状态，不是反问
- 他说"直接做，不要问我" → 听他的
- 做错了立刻改方向，不需要解释为什么错
- 有更好的方案一开始就用，不要因为"已经做了一部分"而将就——他说过"转一个大湾，白干那么多事"

### Step 1: 查询相关记忆
```bash
python ~/.hermes/memory/boshi_memory.py search "<用户消息>" --user lezhi --top-k 5
```
将返回的记忆注入到你的思考中，作为上下文参考。

### Step 2: 回复用户
正常回答，但利用记忆中的信息避免"失忆"。

### Step 3: 存储有价值的记忆
对用户消息中有保留价值的内容进行存储：
```bash
python ~/.hermes/memory/boshi_memory.py add "<记忆内容>" --user lezhi --source chat
```

## 会话入口偏好

乐之的主力入口是**工作台（7681端口）**，原因：
- 网页可以自由复制粘贴大段文字（终端/微信做不到）
- 能看到我的完整工作过程（thinking、tool_calls、content 流式展示）
- 微信看不到工作流，只能看到最终回答

**入口选择优先级：**
1. 工作台（7681）— 第一选择，最顺手
2. 微信 — 方便但看不到工作流
3. TUI（命令行）— 遇到安装向导卡住就放弃，不要纠结

**三者的关系（乐之分得很清楚）：**
- Gateway（8642）— 后台消息中转，不要跟 Dashboard 混为一谈
- Dashboard（9119）— Hermes 自带轻量 Web 面板，一般不常用
- WebUI（8648）— EKKO 做的功能网页前端，独立启动

**长期目标：让伯仕成为独立智能体**
- 所有入口共享同一套记忆，不论在哪个入口聊都能续上
- 工作台就是伯仕的专属界面
- 记忆系统、工作台、运行配置都放在 `~/.hermes/` 下，可整体迁移
- 最终与 Hermes 框架解耦，不依赖其分裂设计

## 乐之的重要偏好

### 第一优先级：直接做，不解释

乐之最烦的就是问方案。所有技术选型、架构决策，自己做。以下是他原话：

- "我不是专业人士，我不懂程序设计和架构，所以，你不需要问我，你只需要考虑怎么实现我的需求"
- "既然有更好的方案，一开始就用啊，转一个大湾，白干那么多事"
- 从不问"要继续吗"——做完了自然呈现

**这意味着：**
- **不要**问"方案A还是方案B好？"
- **不要**问"你觉得怎么实现？"
- **不要**问"要搞哪个？"、"要继续吗？"
- 直接选最优方案，直接做。做错了立刻改，不用解释为什么选错
- 宁可做错再改，也不要问着"你怎么看"
- 如果发现绕路了，果断切方案，别心疼已投入的工时

### 第二优先级：过程可视化

乐之想要在浏览器里看到我的完整工作过程。
工作台 v5 通过 Gateway API SSE 流实现了：
- thinking → 左栏灰色斜体
- tool_calls → 左栏黄色
- content → 右栏流式
- 不开终端也能用（Gateway 后台常驻）
- 微信和浏览器都是同一个伯仕

### 不调 LLM 做记忆提取

⚠️ **重要教训：** 不要用 LLM 从用户消息中"提炼"记忆。直接存原文。
- 原始 mem0 设计有一步 LLM 提取，但这是多余的
- 调用 LLM 做提取增加 10-30 秒延迟
- 乐之明确纠正过：`add` 命令不应该调用 LLM

实现原理：`add` 命令只做向量化（调用 Ollama bge-m3）+ 存储，不做任何文本处理。

## 技术实现

### 向量模型
用 **bge-m3:latest**（Ollama 本地模型，1.2G）。
- 乐之本机已有，无需额外下载
- 1024维，中文语义召回强
- 不要换 nomic-embed-text（老旧）或 qwen3-embedding（过大2.5G）

### 存储格式
纯文件存储，零额外依赖：
- `vectors.npy` — numpy 二进制向量
- `metadata.json` — 记忆文本+时间戳+来源

### 速度预期
- 搜索：~2-3秒（调用 bge-m3 做向量化 + numpy 余弦距离计算）
- 添加：~2-3秒（同样只做向量化）

## 什么内容值得存
- ✅ 用户的偏好、习惯、个人信息
- ✅ 关键决定、项目方向、技术选型
- ✅ 重要的环境配置、路径、账号
- ✅ 项目进度、当前工作上下文
- ❌ 寒暄、客套话、对话开场白
- ❌ 临时的、一次性的事

## 存储格式
每条记忆应该是**完整的中文陈述句**，主语明确。
- 好的例子：`"乐之的终极目标是UE5虚拟寺院"`
- 坏的例子：`"UE5虚拟寺院"`（缺主语）

## 多用户支持
默认使用 `--user lezhi`。如有其他人使用，分别用不同 user_id。

## 已知陷阱

### ❌ 不要问用户选方案
乐之明确说过自己不是专业人士，所以：
- 不要在技术选型上问他"方案 A 还是 B"
- 直接做决定，做了再说
- 做错了立刻改，不用解释为什么错

### ❌ 不要用 LLM 做记忆提取
曾经试过用 qwen3.5-4b-buddhist 从用户消息中提取关键事实，但：
- 增加 10-30 秒延迟
- 提取质量不稳定
- 乐之直接指出"为啥要调用 qwen3.5-4b"
- 修复：直接存原文，跳过 LLM

### ❌ 工作台不要走文件通道
工作台 v5 直接通过 Gateway API 的 `/v1/chat/completions` SSE 流工作。
之前走 inbox/outbox 文件通道的方案（v3/v4）绕了大弯路，乐之明确批评过。
任何时候提到工作台架构，直接用 Gateway SSE 方案，不要再提文件通道。

## 维护命令
```bash
# 查看所有记忆
python ~/.hermes/memory/boshi_memory.py get-all

# 搜索记忆
python ~/.hermes/memory/boshi_memory.py search "<关键词>" --user lezhi
# 调整召回阈值
python ~/.hermes/memory/boshi_memory.py search "<关键词>" --user lezhi --top-k 10

# 统计
python ~/.hermes/memory/boshi_memory.py stats

# 清空记忆（慎用）
python ~/.hermes/memory/boshi_memory.py delete-all
```

## 迁移说明
整个 `~/.hermes/memory/` 目录可以随配置迁移。
换设备时拷走 `.hermes` 目录即可。依赖只有：
- Ollama 运行中（bge-m3 模型已下载）
- Python 环境有 numpy（Hermes venv 已有）
